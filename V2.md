# LaTeX Compiler Service API v2 Technical Design Document

## Summary

We will build v2 of the LaTeX compiler service to support **multi-file compilation** without requiring users to manually create a zip, while keeping the service **stateless, secure, and easy to scale**.

v2 will add:

* A new multi-file compile endpoint (multipart with many files)
* Stronger zip and path security
* Reliable cleanup of temp work directories
* Better log and error parsing
* A full testing suite for correctness and security cases

This service is still a compiler only. Project storage and editor features remain in the Overleaf clone backend.

---

## Goals

* Support compiling a LaTeX project with many files:

  * `main.tex` plus `\input{}`, `\include{}`, images, `.bib`, styles, classes
  * nested folders like `src/main.tex`, `figures/a.png`
* Keep the compiler stateless and scalable:

  * request contains the full project snapshot
  * no database, no S3 fetch, no project IDs
* Improve security:

  * prevent zip slip, symlinks, traversal, macro abuse
  * enforce tight resource limits
* Improve reliability:

  * guaranteed cleanup of temp dirs
  * stable PDF detection
* Add strong automated tests:

  * unit, integration, security, and regression tests

---

## Non-goals

* Not building an Overleaf alternative in this service
* No persistent project file storage
* No collaborative editing features
* No background job system in v2 (async can be a later v3)
* No network access from the compiler runtime

---

## Current v1 Issues to Fix in v2

1. PDF output detection is incorrect for non-`main.tex` main files (PDF name mismatch).
2. Temp work directories can leak and accumulate (cleanup not guaranteed).
3. Zip main-file detection is shallow and can miss nested projects.
4. Size limits and security checks exist but need to be made consistent across inputs.

## Proposed v2 API

### 1) Health

**GET `/health`**

* Return service status and which engines are available.

Response:

```json
{
  "status": "ok",
  "version": "2.0.0",
  "engines": ["pdflatex"]
}
```

---

### 2) Multi-file compile (new)

**POST `/v2/compile/sync`** (multipart/form-data)

Fields:

* `main_file` (string, required)
  Example: `src/main.tex`
* `files` (UploadFile, repeated, required)
  Each file must have a filename that is the relative path inside the project.
* `engine` (string, optional, default `pdflatex`)
* `passes` (int, optional, default 2)
* `return` (string, optional, default `pdf`)

  * `pdf`: return PDF bytes
  * `json`: return JSON with logs and optionally base64 pdf (optional feature)

Success response (default):

* Status: 200
* Content-Type: `application/pdf`
* Headers:

  * `Content-Disposition: attachment; filename="output.pdf"`
  * `X-Compile-Time-Ms`

Failure response:

* Status: 400 for compile errors, 413 for payload too large, 422 for invalid input
* JSON payload:

```json
{
  "status": "error",
  "error_type": "latex_compile_error",
  "message": "Undefined control sequence",
  "errors": ["Undefined control sequence"],
  "warnings": [],
  "log": "...",
  "log_truncated": false
}
```

Notes:

* This endpoint replaces the need for zip for internal editor usage.
* For external callers, we can still support zip upload for convenience.

---

### 3) Zip compile (still supported)

**POST `/v2/compile/zip`** (multipart/form-data)

Fields:

* `file` (zip, required)
* `main_file` (string, required)
* `engine`, `passes`, `return` same as above

Why keep it:

* some clients will prefer sending a zip snapshot
* easier for simple scripts

---

### 4) Validate (optional for v2)

**POST `/v2/compile/validate`** (JSON)

This can remain, but should share the same compiler pipeline code.
It should compile in a temp dir and return only JSON results.

## Request Data Model Rules

### Allowed file paths

For multi-file uploads, each uploaded file’s `filename` is treated as the intended project-relative path.

Rules:

* Must be relative, no leading `/`
* Must not contain `..`
* Must not contain `\` (convert to `/` or reject)
* Must not be empty
* Must be under max path length (ex: 300 chars)
* Must not contain null bytes

Examples:

* OK: `main.tex`
* OK: `src/main.tex`
* Reject: `../main.tex`
* Reject: `/etc/passwd`
* Reject: `C:\windows\...`

### Allowed file types

We should allow most types needed for LaTeX, but block risky ones.

Allowed by default:

* `.tex`, `.bib`, `.bst`, `.cls`, `.sty`
* images: `.png`, `.jpg`, `.jpeg`, `.pdf`
* `.txt`, `.csv` (sometimes used)

Rejected by default:

* executables: `.sh`, `.exe`, `.dll`, `.bin`
* archives inside archives: `.zip`, `.tar`, `.gz` as project members (optional)
* large binary types not needed

This is a whitelist. If you need more later, add explicitly.

## Compilation Pipeline (v2)

### High-level flow

1. Validate request fields and enforce limits.
2. Create temp work directory.
3. Write files into work directory (multi-file) or extract zip safely (zip endpoint).
4. Scan all `.tex`, `.sty`, `.cls` for dangerous macros.
5. Verify `main_file` exists.
6. Run compile (pdflatex, multi-pass).
7. Determine output PDF name based on `main_file`.
8. Read PDF bytes (or stream) and return.
9. Always delete temp work directory.

### Key implementation detail: safe cleanup

Use a single function that wraps the whole compile in `try/finally` and deletes:

* uploaded temp files (if any)
* extracted work directory
* any copied output

No workdir leaks.

## Compilation Strategy

### Engines

v2 baseline: `pdflatex`.

We can add `xelatex` and `lualatex` later, but only if installed and tested.

### Passes

Default 2 passes.

Constraints:

* Min 1, max 5 (hard max)
* Consider lowering max to reduce abuse risk

### Bibliography

For v2:

* Do not automatically run bibtex/biber unless you add a build mode.
* If users need full Overleaf-like behavior, add `latexmk` build mode in v3.

This keeps v2 simpler and safer.

## Security Design

### 1) No shell escape

Always include `-no-shell-escape` in engine command.

### 2) Dangerous macro scan

Scan content of:

* `.tex`
* `.sty`
* `.cls`

Block if any of these appear (exact list can evolve):

* `\write18`, `\immediate\write18`
* `\openout`, `\openin`
* `\newwrite`, `\newread`
* `\input|` pipes

Implementation notes:

* Scan as text with safe decode. If decode fails, treat as suspicious for those file types and reject.

### 3) Zip extraction security (zip endpoint)

* Validate each member path with same path rules above
* Reject absolute paths and any `..`
* Reject symlinks (check zip metadata if available)
* Enforce:

  * max file count
  * max total uncompressed size
  * max single file size

### 4) Resource limits

Hard limits (recommended starting values):

* Max total upload size: 20MB
* Max number of files: 500
* Max compile time: 20s per pass
* Max passes: 5
* Max log size returned: 64KB (truncate and set `log_truncated=true`)

Container limits:

* memory limit configured in deployment
* CPU limit configured in deployment
* no outbound network allowed

### 5) File system safety

* Work directory is isolated and random
* Do not write outside work directory
* When writing files:

  * create parent dirs safely
  * never follow symlinks
  * open with safe flags if possible

### 6) Abuse controls (recommended)

* API key auth for production usage
* Rate limiting per key and per IP
* Optional deny list for repeated abusers

## Error Handling and Response Schema

Standardize all failures:

Common fields:

* `status`: `"error"`
* `error_type`: `"invalid_input" | "payload_too_large" | "latex_compile_error" | "timeout" | "internal"`
* `message`: short human message
* `errors`: parsed errors list (best effort)
* `warnings`: parsed warnings list (best effort)
* `log`: truncated log
* `log_truncated`: boolean

Status codes:

* 400: compile error or bad request
* 413: too large
* 422: invalid fields (bad path, main_file missing, etc)
* 500: internal

Never return Python stack traces.

## Testing Plan

### Test categories

#### A) Unit tests

* Path validation:

  * rejects `..`, absolute paths, backslashes
* File type validation:

  * allows expected extensions
  * rejects dangerous extensions
* Dangerous macro scanning:

  * detect blocked macros in `.tex`, `.sty`, `.cls`
* PDF name selection:

  * `src/main.tex` should map to `src/main.pdf` output name (based on stem)
* Log parsing:

  * extract errors lines starting with `!`
  * extract warnings containing `LaTeX Warning`

#### B) Integration tests

Use real pdflatex in CI if possible, or run in a docker test environment.

Cases:

* Simple multi-file project:

  * `main.tex` inputs `chapters/one.tex`
* Nested main file:

  * main at `src/main.tex`
* Project with image:

  * includes `figures/a.png`
* Project with `.sty` custom package
* Zip endpoint end-to-end compile

#### C) Security tests

* Zip slip attempt:

  * zip contains `../evil.tex`
* Absolute path attempt:

  * `/etc/passwd`
* Symlink in zip (if representable)
* Too many files
* Too large extracted size
* Dangerous macro in `.sty`
* Timeout TeX:

  * infinite loop or very slow compile
  * expect timeout error_type

#### D) Regression tests

Add tests for every bug fixed:

* wrong expected pdf name
* leaked workdir cleanup
* nested `.tex` detection issues

### Test tooling

* pytest for unit and integration tests
* Use FastAPI test client for API tests
* Add fixtures:

  * sample projects stored in `tests/fixtures/projects/*`
  * generated zip fixtures

## Implementation Plan

### Step 1: Refactor compiler into a single pipeline module

Create a single internal function like:

* `compile_project(work_dir, main_file, options) -> result`

And two input adapters:

* `build_workdir_from_multifile_upload(files, work_dir)`
* `build_workdir_from_zip(zip_file, work_dir)`

This prevents duplication and keeps behavior consistent.

### Step 2: Add `/v2/compile/sync` endpoint

* multipart with repeated `files`
* strict path validation
* total size enforcement
* calls shared pipeline

### Step 3: Add `/v2/compile/zip` endpoint

* shared pipeline
* stronger zip security
* require main_file always

### Step 4: Add security hardening and limits

* consistent limit enforcement
* macro scanning expanded to `.sty` and `.cls`
* log truncation

### Step 5: Add test suite

* unit tests first
* integration tests next
* security tests last

### Step 6: Rollout

* Keep v1 endpoints for compatibility short-term
* Deploy v2 in parallel
* Update Overleaf clone backend to use v2 multifile
* After stable, deprecate v1

---

## Acceptance Criteria

v2 is done when:

* Multi-file compile works without zip for nested projects.
* Zip compile still works and is more secure.
* No temp directories leak after requests.
* Dangerous macros are blocked in `.tex`, `.sty`, `.cls`.
* Limits are enforced and tested.
* Test suite covers:

  * at least 10 integration cases
  * at least 10 security cases
  * path validation thoroughly
